
## Frontend ZIP — CSV per-tab + TXT dati campionati (strategia C) — 14/06
Lo ZIP di download del comune (setupDownloadButton in comune.html, generato CLIENT-SIDE con JSZip) ora include, oltre a dati-completi.json + LICENSE + METADATI.csv + README:
- csv/<sezione>_<nodo>.csv per OGNI nodo completo del JSON. Regola di flattening: lista-di-oggetti->record per riga (dict annidati con chiave puntata); dict-di-scalari->chiave,valore; dict-di-dict (2 liv)->riga per chiave; dict profondo (>=3 liv)->split sul 1° livello + formato lungo (liv1,liv2,...,valori); scalari sparsi->_sintesi.csv; kpi_summary->formato lungo sezione,metrica,valore. UTF-8 BOM (\ufeff) per Excel, CRLF, escaping RFC 4180. ~79 CSV per Lecce.
- dati-dettaglio/<x>_COME_SCARICARE.txt per le collezioni CAMPIONATE/non-nel-json: NON si mettono i dati parziali in CSV, si rimanda al file completo gia' esposto via nginx. Decisione troncamento PER-COMUNE (non statica): un nodo e' campionato solo se nel JSON la lista e' piu' corta del totale KPI (es. beni_culturali troncato a Lecce 30/654 -> .txt; completo a Francavilla 1/1 -> CSV). Helper isCampionato() confronta len(lista) vs kpi totale. asia.ateco_dettaglio/catasto/censimento_sezioni: sempre .txt (dettaglio mai nel JSON). Link .txt: /data/anncsu_full, /data/immobili_pa, /data/beni_culturali_full, /data/asia, /data/censimento_full, /data/catasto_full (tutti esposti, vedi commit nginx). NB beni_culturali_full esiste solo per comuni >30 beni -> per gli altri il dato e' completo nel JSON e va in CSV (gestito da isCampionato).
Prerequisito: esposizione nginx delle 20 cartelle /data/<sez>/ (commit deploysimbacruscotto 6264705). Verificato e2e: ZIP Lecce + Francavilla, CSV ben formati, .txt con link raggiungibili.
